# 组件统一后续：全技术债与性能优化总路线图

> 日期：2026-03-19
> 适用范围：`AiiinOB` 当前主干
> 前置条件：`组件统一与技术债治理任务指导书` 主线已完成，前端组件体系已基本统一
> 目标：在组件统一完成之后，继续把当前已知的技术债与性能优化工作系统性收尾

---

## 1. 文档定位

本文件回答的问题不是“如何完成组件统一”，而是：

`在组件统一已经完成之后，还需要做哪些任务，才能把当前已知的技术债和性能优化工作基本做完。`

它覆盖的是：

- 组件统一之后的剩余结构债
- 平台与基础设施层的技术债
- 运行时与可观测性闭环
- 全仓性能优化主项
- 长期维护与清仓任务

这份文档的目标不是承诺“此后永远没有技术债”，而是把当前已知高优先级债务尽量做完，把剩余问题压缩到正常维护规模。

---

## 2. 与前一份文档的边界关系

前一份文档负责：

- 最小组件迁移
- 组件相关技术债
- 阻碍组件统一的状态债、结构债、性能前置问题
- `Options` / `content` 组件最终统一

本文件负责：

- 组件统一之后仍然存在的全仓技术债
- 所有与平台、基础设施、运行时、兼容性、构建、性能主项有关的收尾工作
- 把“UI 统一完成”推进到“工程整体进入长期稳定维护状态”

换句话说：

- 前一份文档完成后，前端组件统一应当完成
- 本文件完成后，当前已知的大部分技术债和性能优化主项应当完成

---

## 3. 总体完成标准

只有当以下条件同时满足时，才可认为“当前已知技术债和性能优化工作基本完成”：

- 前端组件体系统一完成
- 平台与依赖边界清晰，不再存在明显回流
- 主要 orchestrator 已不再构成维护主阻塞
- 状态链与内容脚本启动链显著减重
- 构建与分包优化落地
- 运行时观测、错误上报、同意链路已闭环
- 真实浏览器回归体系完整
- 文档、脚本、目录和残余兼容层已完成一轮清仓

---

## 4. 测试与验证要求

本文件所有阶段都必须同时满足三类验证：

- 自动化验证
- 手动验证
- 浏览器自动操作验证

允许并推荐使用 Chrome DevTools MCP 执行手动测试与浏览器级验证。

Chrome DevTools MCP 可用于：

- 打开扩展页面、站点页面和测试页面
- 操作表单、按钮、下拉框、弹窗、拖拽、键盘快捷键
- 读取 a11y tree / DOM 结构 / console / network
- 检查 Shadow DOM、样式注入、焦点陷阱和状态变化
- 为真实回归保留一致的执行路径

任何涉及以下内容的任务，都不能只依赖单测：

- 扩展页面加载
- background / content / options 跨上下文联动
- Reader / Video / Clipper 主流程
- Shadow DOM 样式桥接
- 视频站和 AI 平台兼容
- Sentry / analytics / consent 真机行为

---

## 5. 总体执行顺序

推荐顺序如下：

1. 阶段 F：平台与依赖边界最终收口
2. 阶段 G：结构债与大文件治理收尾
3. 阶段 H：性能优化主项实施
4. 阶段 I：运行时闭环与观测系统收口
5. 阶段 J：兼容性与真实场景回归
6. 阶段 K：文档、脚本、目录与历史遗留清仓
7. 阶段 L：最终验收与长期维护基线建立

---

## 6. 阶段 F：平台与依赖边界最终收口

### 6.1 目标

把“组件统一之后仍然残留的基础设施层模糊边界”收干净。

### 6.2 必做任务

#### F-1 收紧平台服务保留边界

目标：

- 继续维持并审查 `getPlatformServices()` allowlist
- 不允许新的 listener / pipeline / service / UI 层回流平台调用
- 如仍存在临时兼容路径，明确是否继续保留

重点：

- `src/background/index.ts`
- `src/content/index.ts`
- `src/options/index.ts`
- `src/platform/services.ts`

#### F-2 决定 DI 和 Repository 的长期形态

本阶段必须回答三个问题：

1. 哪些 DI 能力是当前项目长期需要的
2. 哪些 repository 抽象是有必要的
3. 哪些 compatibility adapter 只是历史遗留，应退役

要求：

- 不继续无边界扩展抽象
- 对保留的抽象写明职责
- 对退役的抽象给出迁移计划

#### F-3 清理默认依赖与隐式注入

目标：

- 减少模块级默认状态
- 减少隐式 fallback
- 减少“没传依赖也能跑，但行为不透明”的路径

### 6.3 验收

- 平台调用边界可枚举、可审计
- DI / repository / compatibility 的长期形态清晰
- 新增代码不会轻易重新引入 service locator 式回流
- 通过自动化审计与 Chrome DevTools MCP 辅助回归验证相关主流程

---

## 7. 阶段 G：结构债与大文件治理收尾

### 7.1 目标

把组件统一阶段未完全收完的结构债真正做成“可长期维护”状态。

### 7.2 必做任务

#### G-1 收尾剩余大 orchestrator

重点检查：

- `src/content/video/session.ts`
- `src/content/reader/session.ts`
- `src/content/clipper/components/dialog.ts`
- `src/options/app/bootstrap.ts`
- `src/third_party/ai-chat-exporter/shared/markdown.ts`
- `src/content/video/platforms/bilibiliPlatform.ts`

要求：

- 主文件不再承担过多职责
- 状态、视图、平台、序列化、运行时协调边界清晰

#### G-2 清理重复 normalize / sanitize / clone 逻辑

目标：

- 统一归一化工具来源
- 避免同一类逻辑多处复制
- 减少后续维护时出现“看起来一样，细节不同”的工具函数

#### G-3 收紧类型系统默认基线

重点：

- 继续降低 `any`
- 降低不必要的 `unknown`
- 降低多余 `as`
- 推动 strict 口径向默认基线靠拢

#### G-4 清理事件与生命周期泄漏风险

重点检查：

- `setTimeout`
- `setInterval`
- `addEventListener` / `removeEventListener`
- `ResizeObserver` / `MutationObserver`
- session / panel / dialog 的 dispose 流程

### 7.3 验收

- 主要大文件不再是维护主阻塞
- 重复归一化逻辑明显减少
- 类型系统风险显著下降
- 事件和定时器清理路径清晰
- 对应流程完成自动化回归与 Chrome DevTools MCP 浏览器级验证

---

## 8. 阶段 H：性能优化主项实施

### 8.1 目标

把现有性能评估报告中最有收益的优化项真正落地，而不是停留在分析层。

### 8.2 必做任务

#### H-1 构建与分包优化

重点：

- `scripts/build.mjs`
- 入口最小化
- 让动态 `import()` 产生真实分包收益

目标：

- 降低 `options` 首包
- 降低 `content` 注入开销
- 避免继续把重逻辑打进主包

#### H-2 `Options` 状态链性能优化

重点：

- `StateManager`
- `optionsStore`

目标：

- 降低深拷贝
- 降低序列化比较
- 降低全量广播
- 提高交互流畅度

#### H-3 content 启动链懒化

重点：

- `src/content/index.ts`
- `src/content/runtime/bootstrapRuntime.ts`

目标：

- Reader / Video / Prompt / Extractor 只在需要时初始化
- 降低普通网页常驻成本

#### H-4 Reader / Article / Video 热点优化

重点：

- `reader/utils/markdownBuilder.ts`
- `extractors/articleExtractor.ts`
- `video/platforms/bilibiliPlatform.ts`
- `video/session*`

目标：

- 减少重复 DOM 查询
- 减少重复序列化与转换
- 降低高变更页面的 CPU 抖动

### 8.3 验收

- 关键包体有可见下降或结构性改善
- `Options` 交互热路径减重
- content 初始化更轻
- 热点路径有 profile 或行为对比结果
- 使用 Chrome DevTools MCP 或等价真实浏览器方案完成至少一轮优化前后对比

---

## 9. 阶段 I：运行时闭环与观测系统收口

### 9.1 目标

让项目具备真正上线后的可观测性和问题定位能力。

### 9.2 必做任务

#### I-1 Sentry/analytics 真机联调

需要完成：

- DSN rollout
- release / environment 注入验证
- consent 链路联调
- 错误采样与启用条件验证

#### I-2 错误链路与上报策略收口

目标：

- background / content / options 的错误上报路径清晰
- 不重复、不漏报、不误报
- 隐私边界和同意状态可解释

#### I-3 运行时异常回放与定位能力验证

要求：

- 至少构造几类真实失败场景
- 验证 console、network、error reporting、UI 提示是否一致
- 必要时使用 Chrome DevTools MCP 检查 network request、console message、页面状态

### 9.3 验收

- Sentry / analytics / consent 真机可验证
- 真实异常路径能被定位
- 运行时观测链路具备最小闭环

---

## 10. 阶段 J：兼容性与真实场景回归

### 10.1 目标

把“代码逻辑正确”推进到“真实网站、真实浏览器、真实页面条件下也稳定”。

### 10.2 必做任务

#### J-1 浏览器兼容回归

至少覆盖：

- Chrome
- Firefox

重点检查：

- 扩展加载
- options 页面
- content script 注入
- Shadow DOM 样式桥接
- 通知、消息、tabs、storage 等 API 行为

#### J-2 站点兼容回归

至少覆盖：

- 常见网页文章页
- AI 平台页面
- 视频站页面
- Shadow DOM 或复杂 DOM 页面

#### J-3 关键交互回归

包括：

- 剪藏
- 片段标注
- 阅读会话
- 视频片段
- vault 路由
- YAML 配置
- 语言切换
- 键盘操作与焦点陷阱

#### J-4 手动测试执行要求

本阶段必须明确允许执行者：

- 使用 Chrome DevTools MCP 自行打开测试页面
- 自行操作扩展 UI
- 自行检查 DOM、样式、console、network
- 记录真实回归结果

对于难以自动化覆盖的功能，Chrome DevTools MCP 应被视为标准验证工具之一，而不是可选项。

### 10.3 验收

- 核心场景跨浏览器通过
- 关键站点无明显阻塞问题
- 手动回归覆盖面完整
- Chrome DevTools MCP 执行记录可复现关键验证步骤

---

## 11. 阶段 K：文档、脚本、目录与历史遗留清仓

### 11.1 目标

把“已经做完但还留着历史噪音”的部分收掉，避免后续理解成本继续偏高。

### 11.2 必做任务

#### K-1 文档真值统一

目标：

- 更新过时文档
- 归档旧路线图和旧状态报告
- 确保主入口文档与当前代码一致

#### K-2 脚本职责整理

目标：

- 清理重复脚本
- 统一脚本命名和职责划分
- 减少只服务历史批次的残留脚本

#### K-3 目录与导出边界整理

包括：

- 目录层级清理
- barrel/export 边界
- 参考文件、临时文件、归档目录清理

#### K-4 i18n / 测试 / coverage 尾项

包括：

- i18n 生成化推进
- coverage 薄弱点补测
- 进一步统一测试入口与执行说明

### 11.3 验收

- 文档入口清晰且可信
- 脚本和目录结构可解释
- 旧批次历史噪音明显减少
- 新开发者理解成本下降

---

## 12. 阶段 L：最终验收与长期维护基线建立

### 12.1 目标

为“当前已知高优先级技术债基本清空”建立正式结论和长期守门机制。

### 12.2 必做任务

#### L-1 形成最终验收报告

报告应覆盖：

- 已关闭的技术债清单
- 已落地的性能优化清单
- 仍保留但不再高优先级的问题
- 不在本轮处理范围内的事项

#### L-2 建立长期守门

建议守门内容：

- 平台调用 allowlist 审计
- 深层导入审计
- 新增基础控件手写实现审计
- 构建产物体积监控
- 核心热路径回归检查

#### L-3 建立长期维护议题池

这部分不再叫“主线债务”，而是进入长期维护：

- 新站点适配
- 新 UI helper
- i18n 扩展
- 可观测性策略微调
- 性能微优化

### 12.3 最终完成标准

当以下条件满足时，可以宣布“当前已知高优先级技术债与性能优化工作基本完成”：

- 前端组件已统一
- 平台与依赖边界已稳定
- 主要结构债已收口
- 性能主项已落地
- 运行时闭环已完成
- 真实浏览器验证完整
- 文档与工程入口已统一
- 剩余事项已降级为长期维护 backlog

---

## 13. 不在本轮承诺范围内的事项

即使本文件全部完成，也不意味着以下事情永久结束：

- 新增站点适配不会再带来新债务
- 未来不会出现新的性能热点
- 所有实验性功能都能永久保持最佳架构形态

本文件的承诺边界是：

把当前已知、已暴露、已形成结构性影响的高优先级问题系统性收尾。

---

## 14. 推荐落地方式

如果按单人推进：

1. 先完成阶段 F
2. 再完成阶段 G
3. 再完成阶段 H
4. 然后做阶段 I 与 J
5. 最后做阶段 K 与 L

如果按多人推进：

- 一条线负责平台与结构债
- 一条线负责性能主项
- 一条线负责浏览器回归与观测闭环

但要求：

- 共享边界先定清楚
- 手动测试口径统一
- Chrome DevTools MCP 的操作路径和回归脚本有记录可复用

---

## 15. 最终建议

当组件统一完成之后，项目已经过了“最容易看出问题”的阶段。

后面的工作更难，因为它们多数不再是视觉混乱，而是：

- 边界是否真的干净
- 性能是否真的改善
- 运行时是否真的可观测
- 真实浏览器里是否真的稳定

因此，这一阶段的价值不在于“看起来更整齐”，而在于：

把工程从“高质量但仍有大量隐性复杂度”，
推进到“结构可维护、性能可接受、运行时可定位、长期可持续演进”。

这才是当前项目从成熟走向稳定的最后一段路。
