# 综合执行优先级计划（CSS / 设计系统 / 技术债务）

> **更新日期**：2026-03-13  
> **用途**：将 CSS / Tailwind 收口、设计系统迁移、`docs/债务.md` 中的技术债务合并为一份统一执行顺序，供执行人按阶段推进，后续再由审阅人复核  
> **判定口径**：以当前仓库代码真值、`STYLE-MIXING-AUDIT.md`、`TAILWIND-MIGRATION-STATUS.md`、`PENDING-TASKS.md` 与 `docs/债务.md` 的交叉判断为准

---

## Summary

当前项目的剩余工作不能拆成彼此独立的三摊来做：

- **CSS / Tailwind 主线** 决定哪些样式层要保留、哪些还要继续迁移
- **设计系统迁移文档** 决定后续执行人会按照什么口径推进
- **技术债务治理** 决定后续代码是否还能继续稳定演进

这三者的逻辑关系是：

1. **先完成改代码前的迁移准备**
2. **再按“完全迁移”目标推进样式主线**
3. **再统一设计系统与架构文档**
4. **再处理真正阻塞后续演进的 P0 技术债务**
5. **最后做中低优先级整合与清理**

如果顺序反过来，就会出现：

- 在迁移目标未定前，提前做 token / Tailwind config 合并，导致返工
- 在设计系统文档未校正前，别人按过时计划继续执行
- 在平台依赖 / 全局状态 / 超大文件未收敛前，继续堆叠 UI 或样式工作，导致复杂度继续上升

## Priority Model

### P0：改代码前必须先完成的准备工作

这组任务不一定代码量最大，但会决定后面所有工作是否返工：

- 样式迁移目标从“边界收口”明确升级为“完全迁移”
- 为各模块写清楚迁移终态、禁止保留项、可阶段性保留项
- 校正 Tailwind / 设计系统文档，让后续执行人不会按“接受混用”口径做事

### P1：阻塞后续演进的核心技术债务

这组任务一旦继续拖延，会持续放大后续所有模块改动成本：

- `getPlatformServices()` 依赖清理
- 全局 `window.__aiob*` 状态污染收敛
- 运行时依赖注入检查模式简化
- 超大文件拆分

### P2：体系收敛与维护性优化

这组任务重要，但应建立在前两组完成之后：

- Token 双轨口径收敛
- Tailwind 配置碎片化收敛
- 路径别名、目录深度、barrel 整理
- 构建脚本整理
- TODO / 薄层维护项清理

## Fixed Execution Order

### Phase 1：先完成改代码前的样式迁移准备

**目标**：在动代码前，先把样式目标、模块终态、验收标准和迁移禁区写死。

**原因**：

- `STYLE-MIXING-AUDIT.md` 已经确认项目仍是混合样式架构
- 当前最大的未定项不是 `Clipper` / `Video`，而是 `Onboarding`
- 如果在终态未定前就推进迁移，极容易返工

**执行项**：

1. 明确总目标：
   - 样式边界目标是**完全迁移**
   - 不以“接受长期混用”作为终态
2. 为以下模块写出迁移终态定义：
   - `Options`
   - `Onboarding`
   - `Clipper`
   - `Reader`
   - `Video`
   - `SupportPrompt`
3. 为每个模块写清楚：
   - 目标完成后允许存在的样式入口
   - 必须删除或退出主路径的 legacy CSS / inline style / runtime 注入
   - 允许阶段性过渡、但最终必须消失的保留项
4. 输出迁移前检查清单：
   - 文件级范围
   - 模块级完成标准
   - 回归验证清单
5. 单独处理 `Onboarding`：
   - 不再停留在“是否接受混用”的判断题
   - 直接定义为**必须纳入完全迁移范围**

**完成标准**：

- 所有模块都拥有明确的“完全迁移终态”定义
- 后续执行人不需要再猜哪些混用会被长期保留
- 改代码前的范围、终态、禁区、验收标准全部成文

---

### Phase 2：开始执行样式完全迁移主线

**目标**：按 Phase 1 定义的终态，逐模块把样式从混用收敛到完全迁移。

**固定顺序建议**：

1. `Onboarding`
2. `Options`
3. `SupportPrompt`
4. `Video`
5. `Clipper`
6. `Reader`

**原因**：

- `Onboarding` 是当前最明显未收口点，应最先清掉
- `Options` 会影响 token、组件层和设计系统口径
- `SupportPrompt` / `Video` / `Clipper` / `Reader` 共享 content 侧样式架构，应按依赖链逐步收

**执行项**：

1. 清理 inline `<style>` 与运行时硬编码样式
2. 把仍在主路径中的 legacy CSS 迁出
3. 逐步缩减 runtime 注入仅保留临时过渡能力，最终退出主路径
4. 更新模块测试与验证口径，保证迁移过程中行为不变

**完成标准**：

- 目标模块不再以混用作为正式架构
- 主路径样式入口与 Phase 1 定义一致
- “完全迁移”从文档目标变成代码事实

### Phase 3：统一设计系统 / Tailwind / 总体计划文档

**目标**：让后续执行人看到的所有主文档都使用同一套真实口径。

**原因**：

- 现在 Tailwind 活跃文档已基本改写完成，但总入口与上层计划还未全部同步
- 如果不先做文档统一，后续别人会按旧路线继续做错事

**执行项**：

1. 回写 `STAGE3-IMPLEMENTATION-PLAN.md`
   - 重写进度跟踪
   - 对齐 Tailwind 主线已收口的事实
   - 明确 Stage 3 剩余真实活跃项
2. 回写 `ARCHITECTURE-REFACTOR-PLAN.md`
   - 校正 Month 1-4 状态
   - 明确哪些是已完成、哪些是未完成、哪些已改口径
   - 把“样式架构仍为混合架构”的事实纳入说明
3. 最后统一回写总入口：
   - `README.md`
   - `PENDING-TASKS.md`
   - `ARCHIVE-INDEX.md`

**完成标准**：

- 不再出现同一事项在不同文档中状态冲突
- `251126-design-system-poc` 目录下的主入口口径统一

### Phase 4：开始处理真正的 P0 技术债务

**目标**：处理那些已经在持续抬高复杂度、阻塞后续演进的核心结构问题。

**这一步一定要晚于 Phase 1 / 2 / 3**，因为它会影响后续设计系统和模块拆分的方式。

#### 3.1 第一刀：清理平台服务依赖边界

**优先项**：

- 清理 `getPlatformServices()` 的跨层滥用
- 推进 Repository / 显式依赖替代

**原因**：

- 这是当前最核心的分层问题之一
- 不先收这个口，后面的 Session / UI / store 拆分都会很痛苦

**涉及重点**：

- `content` 入口层
- `sessionDependencies`
- `shared` 层误用平台服务的位置

#### 3.2 第二刀：收敛全局状态与运行时依赖初始化

**优先项**：

- 清理 `window.__aiob*` 全局状态污染
- 清理运行时依赖检查型初始化模式

**原因**：

- 当前状态管理分散、难测、难追踪
- `VideoSession` / `ReaderSession` / `Clipper` 等复杂模块的拆分，会被这些模式持续阻塞

#### 3.3 第三刀：拆超大文件

**固定顺序建议**：

1. `src/content/video/session.ts`
2. `src/options/components/controls/yamlConfigTable.ts`
3. `src/content/clipper/components/dialog.ts`
4. `src/content/ui/supportPrompt.ts`
5. `src/content/video/ui/panel.ts`

**原因**：

- 这些文件正好处在“平台服务 + UI + 状态 + 样式”耦合最重的位置
- 它们不拆，其他债务很难真落地

**完成标准**：

- 依赖边界更清晰
- 类职责单一
- 后续测试、样式与功能改动成本下降

### Phase 5：前端体系与样式配置收敛

**目标**：在样式完全迁移主线完成、核心结构初步收敛之后，再做样式体系层面的整合。

#### 4.1 Token 口径收敛

**做什么**：

- 以“减少双轨 token 体系”为目标
- 在不破坏已完成迁移结果的前提下，推进 token 源头合并或强约束别名层
- 只保留明确有必要的桥接层

#### 4.2 Tailwind 配置收敛

**做什么**：

- 评估 `tailwind.config.cjs`、`tailwind.config.global.cjs`、`tailwind.config.clipper.cjs`、`tailwind.config.video.cjs` 的可合并边界
- 能共享的抽共享层
- 必须拆开的保留模块化配置

**原则**：

- 目标是降低维护成本
- 不是为了形式上只保留一个配置文件

#### 4.3 CSS / 组件保留层最终收口

**做什么**：

- 确认 `aob-options.css` 的长期职责
- 确认 `highlight-themes.css` 的长期职责
- 若 `Onboarding` 决定继续迁移，再在这一阶段完成代码侧收口

**目标**：把不会立即阻塞主线，但会影响长期维护效率的问题系统整理掉。

**执行项**：

**注意**：

- 这些项不应插队到 Phase 1-4 前面

## Why This Order

### 先准备完全迁移，再做代码迁移，再做技术债务

因为当前最容易误导后续工作的，不是“代码还不够优雅”，而是：

- 样式迁移终态还没被明确写死
- 设计系统文档仍有历史口径残留

如果不先处理这两件事：

- 后面做 token 合并会返工
- 后面做 Tailwind config 收敛会返工
- 后面别人按旧文档继续推进会继续制造新债

### 先处理依赖边界，再拆大文件

因为超大文件很多本质上是**依赖边界混乱**的结果。

如果不先收 `getPlatformServices()`、全局状态、运行时依赖注入模式：

- 文件即使拆了，也可能只是把复杂度分散到多个文件
- 不会真正提升可维护性

### 把 token / Tailwind 配置合并放到后面

因为这类工作属于**体系收敛**，而不是当前最关键的**方向判定**。

只有在完全迁移主线、模块职责、依赖边界比较稳定之后，做这件事才不会来回返工。
