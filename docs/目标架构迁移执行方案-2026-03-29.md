## 0. 迁移前必须先解决的债务（Pre-flight）

在正式进入 `M1-M9` 之前，建议先处理一组**低爆炸半径、但会显著影响迁移判断和执行效率的前置债务**。

这些问题的共同特点是：

- 不需要等待目标架构搭好才能修
- 不先处理会持续干扰“什么是正式入口、什么是遗留产物、什么是真实状态”的判断
- 修复成本相对低，但能明显降低后续迁移噪音

需要强调：

- **不是所有高优先级债务都应该前置解决**
- 像“运行时依赖注入复杂”“直接 DOM 操作与业务逻辑混杂”“组件相关大文件拆分”这类问题，虽然优先级高，但更适合在 `M3-M7` 的正式架构迁移中一并解决
- 迁移前真正应该优先处理的，是那些会污染迁移基线、制造入口歧义、放大理解成本的债务

### Milestone

`M0：迁移前置债务清理，确保目标架构迁移在干净基线上启动`

### 0.1 前置债务范围

本阶段建议优先处理以下真实存在的债务：

1. 残留全局状态声明仍存在，容易继续误导新代码走 `window` 全局状态路线
2. 历史样式 / 预览 / 临时资产仍位于主源码邻近区域，容易与正式设计系统入口混淆
3. 正式入口、遗留入口、归档入口的文档边界尚不够强约束，容易在迁移中继续引用错误入口

这些问题都属于：

- 当前真实存在
- 修复成本低于主架构债务
- 适合在实施 `M1` 之前先清理

### 0.2 必做任务

#### M0.1 清理残留全局状态声明

目标：

- 清理不再被真实运行时代码使用的 `window.__aiob*` 全局状态声明
- 禁止后续迁移工作继续把 UI/session 状态回流到 `window` 全局命名空间

当前需要重点核查的残留：

- `src/env.d.ts`
  - `__aiobReaderActive`
  - `__aiobReaderController`

修复方案：

- 若已无运行时引用，直接删除残留声明
- 若仍有极少量兼容用途，必须：
  - 在注释中标记为兼容层
  - 写明退役条件
  - 禁止新增生产路径引用
- 在目标架构文档中明确：session/UI state 只能通过 module state、controller state、host adapter state 或 repository/store 传递

验收标准：

- `rg "__aiobReaderActive|__aiobReaderController" src` 结果为 0，或只剩明确标注的兼容层文件
- `npm run typecheck:app` 通过
- `npm run typecheck:tests` 通过
- `content` 相关 harness / 单测无回归

#### M0.2 归档或隔离历史样式与预览大文件

目标：

- 移除“主源码目录中仍混有大体积历史样式/预览/临时建议文件”的噪音
- 在正式开始 `foundation/tokens`、`primitives/layout`、`hosts/shadow` 迁移前，先确保正式样式入口不再与历史文件混淆

当前应优先处理的文件：

- `src/options/obsidian-clipper-style.css`
- `src/options/obsidian-hybrid-preview.html`
- `src/options/optionuicsssuggest.md`

修复方案：

- 对每个文件做分类：
  - 正式生产依赖
  - 参考预览资产
  - 历史临时文件
- 非正式生产依赖文件应：
  - 移入 `docs/archive/`、`docs/reference-fixtures/`、`trash/` 或专门的 preview/archive 目录

- 对于保留文件，必须在文件头写明：
  - 当前用途
  - 是否参与生产构建
  - 后续归档/删除计划
- 补一条轻量审计：禁止正式入口继续引用这些归档资产

验收标准：

- 上述文件已完成分类，并移出易被误认为正式入口的主路径，或全部带有明确归档说明
- `rg "obsidian-clipper-style|obsidian-hybrid-preview|optionuicsssuggest" src docs` 结果能够清楚区分“正式引用”和“归档引用”

验收标准：

## 1. 文档定位

本方案不是一次性大重写计划，而是**在当前代码真值基础上的渐进式迁移路线**。

本方案解决的问题包括：

- 组件 API 与交互契约未完全统一
- `options` / `content` / `shadow` 三类宿主差异没有独立抽象层
- 基础原语、布局模式和业务组件分层不够清晰
- 部分状态联动、样式宿主、图标策略、性能热点仍然耦合在 feature 内
- 当前守门偏功能正确，缺少“目标架构持续收敛”的执行路径

本方案**不追求**：

- 所有 UI 代码在同一阶段全部迁移完成
- 所有 feature 共用同一份组件实现
- 为了统一而引入重框架重写现有 UI

本方案**追求**：

1. 设计 Token 与交互契约统一
2. 基础原语统一为单一 API 语义
3. 样式宿主与运行时宿主差异显式分层
4. 业务组件与 feature 流程边界清晰
5. 迁移过程中始终可构建、可测试、可回退

## 2. 当前状态与目标状态

## 2.1 当前真实状态

当前代码可概括为：

- `Options` 已有较完整组件层：`BaseComponent`、`BaseSection`、`Daisy*`、`OptionsLayout`
- `content` 侧已有 `content/shared/daisy/*`，但 `Clipper` / `Reader` / `Video` 仍存在较多手写 DOM 与宿主特化代码
- `OptionsLayout.ts` 与 `ContentLayout.ts`、`DaisyButton.ts` 与 `ContentDaisyButton.ts` 已出现“语义相似、实现分叉”的重复
- `design-tokens.css` 已是主 token 真值源，但样式宿主与样式注入仍偏场景化
- `OptionsStateManager` / `optionsStore` 已较成熟，但跨区块协作与部分业务控件仍有 controller / registry 式组织
- 构建、分包、测试和审计基础较完整，但“目标架构是否持续收敛”尚无单独的执行基线

## 2.2 目标架构

### 统一程度目标

- `domains`：只抽真正稳定复用的领域组件
- `features`：统一规范，不强求统一实现

## 3. 总体迁移原则

### 原则 1：先统一 contract，再统一实现

优先统一：props、行为、a11y、命名、token、测试契约；不要一开始追求单实现。

### 原则 2：先抽宿主差异，再谈全仓复用

`options` 与 `content/shadow` 的根差异不在按钮长相，而在挂载、样式注入、生命周期和焦点管理。

### 原则 3：基础原语和业务组件分离

`Button`、`Dialog`、`SettingRow` 等进 `ui`；`VaultRouter`、`YamlConfigTable` 等先作为 `domains` 或保留在 feature 层，等复用足够稳定再上收。

### 原则 4：每个 Milestone 都必须同时完成三件事

- 代码落地
- 守门脚本或测试补齐
- 真实浏览器验证完成

### 原则 5：旧入口只允许“短期兼容”，不允许长期双轨

每个阶段都要写清：

- 新入口是什么
- 旧入口还能存在多久
- 什么条件下删除旧入口

## M2：统一 foundation 层

### 目标

把最底层的公共真值源和横切能力抽出来，形成 `src/ui/foundation`，为后续 primitives 和 hosts 提供稳定地基。

### 必做任务

- `M2.1` 建立 `src/ui/foundation/tokens`，明确 `design-tokens.css` 的单一真值入口与导出路径
- `M2.2` 建立 `src/ui/foundation/icons`：
  - 统一图标白名单
  - 禁止全量 `lucide` 对象导入
  - 提供统一 icon registry / helper
- `M2.3` 建立 `src/ui/foundation/a11y`：
  - 焦点陷阱 contract
  - `aria-invalid` / `aria-busy` / `aria-describedby` 约定
- `M2.4` 建立 `src/ui/foundation/style-host`：
  - 普通 DOM host
  - Shadow DOM host
  - managed stylesheet bridge
- `M2.5` 建立 `src/ui/foundation/types`：
  - 通用 variant / size / dismiss / mount lifecycle 类型

### 迁移重点

- 把现有：
  - `src/styles/design-tokens.css`
  - `src/shared/utils/iconHelpers.ts`
  - `src/shared/shadowStyleBridge*`
  - `FocusTrapController` 相关 contract
    抽成 foundation 层的明确入口

### 验收标准

- `foundation` 目录存在且包含 token、icons、a11y、style-host、types 五类入口
- `Daisy*` 与 `ContentDaisy*` 不再直接各自维护分叉 icon 规则
- 新旧样式宿主都能从统一 style-host contract 进入
- `rg "import \{ icons \} from 'lucide'" src` 结果减少到 foundation allowlist 或为 0
- 自动化验证通过：
  - `npm run typecheck:app`
  - `npm run lint -- --quiet`
  - `npm run audit:imports:report`
  - `npm run audit:design-tokens:report`
- 真实浏览器验证完成：
  - options 页 light/dark token 切换正常
  - content/shadow UI 样式注入正常
  - console 无新增样式注入错误

## M3：统一 primitives contract

### 目标

建立全项目唯一的基础组件语义层，统一 props、a11y、状态语义、交互契约与测试契约。

### 必做任务

- `M3.1` 在 `src/ui/primitives` 建立以下原语入口：
  - button
  - input
  - select
  - checkbox
  - textarea
  - toggle
  - badge
  - alert
  - dialog
  - panel
  - layout
- `M3.2` 为每种 primitive 定义统一 contract：
  - props
  - variant
  - size
  - loading/disabled/error
  - aria 规则
  - keyboard / dismiss 行为
- `M3.3` 建立 primitives contract 测试：
  - unit
  - interaction harness
  - a11y snapshot
- `M3.4` 明确“单一语义、多宿主适配”的落地规则：
  - 可以有 `options` adapter
  - 可以有 `content` adapter
  - 不允许再有无命名空间的第三套实现

### 当前代码建议映射

- `DaisyButton.ts` / `ContentDaisyButton.ts` → `ui/primitives/button/*`
- `DaisyDialog.ts` / `ContentDaisyDialog.ts` → `ui/primitives/dialog/*`
- `OptionsLayout.ts` / `ContentLayout.ts` → `ui/primitives/layout/*`

### 验收标准

- 每个 primitive 都有单独目录、类型与测试入口
- `Options` 和 `content` 的基础按钮/对话框/布局类不再各自定义独立语义字段
- `audit:interaction-contract:report` 能直接覆盖 primitives contract
- 自动化验证通过：
  - `npm run typecheck:app`
  - `npm run lint -- --quiet`
  - `npm run audit:interaction-contract:report`
  - 对应 primitives unit tests
- 真实浏览器验证完成：
  - shared interaction harness 中，button/input/dialog contract 全部通过

## M4：建立 patterns 与 hosts 分层

### 目标

把“基础原语”之上的组合模式和“宿主差异”分开，避免 feature 直接面向低层 DOM 与样式宿主细节编程。

### 必做任务

- `M4.1` 建立 `src/ui/patterns`：
  - form-field
  - setting-row
  - section-shell
  - message-block
  - confirm-flow
- `M4.2` 建立 `src/ui/hosts/options`
- `M4.3` 建立 `src/ui/hosts/content`
- `M4.4` 建立 `src/ui/hosts/shadow`
- `M4.5` 统一 mount/update/destroy contract，至少对 dialog/panel/form shell 生效

### 推荐落地

- `BaseSection` 的 header/body/chrome → `patterns/section-shell`
- `OptionsLayout` 的 row/panel/action row → `patterns/setting-row` + `primitives/layout`
- dialog focus/backdrop/escape 行为 → `patterns/confirm-flow` + `primitives/dialog`
- Shadow DOM style bridge → `hosts/shadow`

### 验收标准

- `patterns` 与 `hosts` 目录存在，且 feature 层可消费
- `BaseSection` 与 `ContentDaisyDialog` 不再直接承担宿主实现细节和模式实现的双重职责
- 至少两个现有 feature 已完成 host/pattern 接入示范（建议：Options shell + Clipper dialog）
- 自动化验证通过：
  - `npm run typecheck:app`
  - `npm run lint -- --quiet`
  - `npm run audit:repository-composition:report`
- 真实浏览器验证完成：
  - Options section shell 正常
  - content/shadow dialog 焦点管理和关闭行为正常

## M5：完成 Options 侧迁移

### 目标

让 `Options` 成为目标架构下的第一块完整示范区：基础原语、模式层、宿主层、状态边界与领域组件组织都完成迁移。

- `M5.5` 收紧 `Options` 状态边界：
  - 基础组件只吃 props
  - 业务 section / controller 才接 store

### 优先迁移对象

- `RestSection`
- `RoutingSection`
- `TemplatesSection`
- `YamlConfigSection`
- `PrivacySection`

## 9. 完成定义（Definition of Done）

本方案完成时，项目应达到以下状态：

- 用户层面：整个产品像一套 UI 系统，而不是多个拼接页面
- 开发层面：团队新增 UI 时只需要学习一套基础规则
- 架构层面：宿主差异、基础原语、模式层、领域层、功能层边界清晰
- 维护层面：重复组件实现不再回流，旧入口不会反复复活
- 性能层面：共享 UI 层不会制造新的首屏和包体负担
- 测试层面：统一架构既能自动验证，也能真实浏览器验证
