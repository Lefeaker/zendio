# 组件统一与技术债治理任务指导书

> 日期：2026-03-19
> 适用范围：`AiiinOB` 当前主干
> 目标：从当前“局部已收口、整体仍混用”的状态，推进到“组件体系最终统一完成”
> 使用方式：本文件作为执行手册、排期依据和阶段验收基线

---

## 1. 文档定位

这不是一份单纯的“组件迁移清单”，也不是一份纯技术债文档。

它覆盖的是从现在开始，到“最终统一组件体系完成”为止的完整任务链，包含：

- 最小必要组件迁移
- 与组件收口直接相关的技术债修复
- 架构和状态链治理
- 性能优化前置条件建设
- 最终全量组件统一与旧实现退役

本文件的原则是：

1. 不先做大而全重构
2. 先止血，再收口，再统一，再做审美级整理
3. 组件统一必须与技术债治理协同推进，而不是彼此割裂

---

## 1.1 测试与验证原则

本文件中的每个阶段都不仅要求“代码改完”，还要求“验证完成”。

验证分为三层：

- 自动化验证：`typecheck`、`lint`、`unit`、`e2e`、`visual`
- 手动验证：真实页面、真实交互、真实浏览器上下文下的人工回归
- 浏览器自动操作验证：可使用 Chrome DevTools MCP 自行打开页面、操作界面、检查 DOM、样式、事件、网络与 console

执行要求：

- 凡是涉及 UI、交互、Shadow DOM、拖拽、弹窗、键盘操作、布局、站点兼容的改动，不能只跑单测
- 必须补对应手动测试
- 手动测试允许并鼓励通过 Chrome DevTools MCP 执行，以减少重复人工操作并保留更稳定的验证路径

推荐将 Chrome DevTools MCP 用于以下场景：

- 打开 `Options` 页面并逐项验证 section
- 验证 `Reader` / `Video` / `Clipper` 的真实 DOM 行为
- 检查按钮、表格、输入框、弹窗、焦点管理和键盘交互
- 检查 Shadow DOM 内样式注入、class 变化和可见性
- 检查 console error、network request、事件触发和渲染结构

最小验证口径：

- 代码改动完成
- 自动化验证通过
- 至少一轮 Chrome DevTools MCP 或等价真实浏览器手动验证完成
- 验证结论回写到任务记录或阶段报告

## 2. 当前真实状态

基于当前仓库、现有文档和代码抽样，当前前端不是“没有设计系统”，而是“设计系统已建立，但未完全吃透”。

当前事实：

- `Options` 已有 `DaisyButton`、`DaisyInput`、`DaisySelect`、`DaisyCheckbox`、`DaisyTable`、`DaisyDialog` 等组件层
- 仍并存 `DaisyUIHelpers.ts` 这类工厂函数式封装
- 多个 section / control 仍直接使用 `document.createElement()` 与手写 Daisy/Tailwind class
- `content` 侧有 `content/shared/daisy/*`，但 `Reader` / `Video` / `Clipper` 仍有较多手写 UI
- 样式侧已进入 Tailwind + DaisyUI 主导，但仍保留 token 层与受控 Shadow DOM bridge

这意味着当前问题不是“缺组件”，而是：

1. 组件入口不唯一
2. 同类控件写法不唯一
3. UI 结构、状态、事件、DOM 构建边界未完全统一
4. 若继续一边加功能一边混用，后续维护成本会继续上升

---

## 3. 总体策略

整个执行过程分为 5 个阶段：

1. 阶段 A：冻结边界与最小组件统一
2. 阶段 B：组件相关技术债与结构债修复
3. 阶段 C：性能优化前置治理
4. 阶段 D：全量组件统一
5. 阶段 E：最终收口与旧体系退役

总原则：

- 阶段 A 必须立即开始
- 阶段 B 与阶段 A 紧密耦合，应交错推进
- 阶段 C 不应早于状态链和组件边界稳定
- 阶段 D 是追求工程美感的主战场，但必须建立在前 3 阶段基础上
- 阶段 E 负责把“看起来统一”变成“代码层面真正统一”

---

## 4. 执行顺序总览

推荐顺序如下：

1. 冻结组件入口与新增代码约束
2. 统一基础控件写法
3. 清理 `Options` 高混用区域
4. 修复与组件迁移强耦合的技术债
5. 稳定状态链与大文件边界
6. 再做性能优化主项
7. 再做 `content` / `reader` / `video` 全量组件统一
8. 删除旧 helper、旧工厂、旧写法残留
9. 做统一验收与设计系统闭环

---

## 5. 阶段 A：冻结边界与最小组件统一

### 5.1 阶段目标

目标不是“立刻把所有页面重写一遍”，而是先建立唯一方向，阻止继续恶化。

本阶段完成后，应实现：

- 新代码不再新增新的组件混用方式
- 同类基础控件有唯一推荐入口
- `Options` 主路径先完成最小统一

### 5.2 必做任务

#### A-1 建立组件使用约束

产出：

- 在开发规范中写清楚组件优先级
- 明确以下约束：
  - 按钮优先用 `DaisyButton`
  - 输入框优先用 `DaisyInput`
  - 选择框优先用 `DaisySelect`
  - 复选框优先用 `DaisyCheckbox`
  - 表格优先用 `DaisyTable`
  - 对话框优先用 `DaisyDialog` / `ContentDaisyDialog`
- 禁止在新增业务代码中直接手写同类基础控件

建议动作：

- 更新 `src/options/components/README.md`
- 增加代码审查清单项
- 若条件允许，增加 ESLint/脚本级轻量守门

#### A-2 统一组件入口

目标：

- 确立 `src/options/components/shared/*` 为 `Options` 唯一基础组件源
- 确立 `src/content/shared/daisy/*` 为 `content` 基础视觉组件源

必须处理：

- 评估并逐步退役 `src/options/components/shared/DaisyUIHelpers.ts`
- 不再新增基于 `createButton()` 的新代码
- 对 `DaisyUIHelpers` 仅保留短期兼容用途

#### A-3 最小基础组件补齐

在开始大规模迁移前，先补齐最常用的缺口，避免迁移时反复手写：

- `DaisyCheckbox` 能覆盖当前 checkbox 主要场景
- `DaisySelect` 能覆盖当前 select 主要场景
- `DaisyTable` 能覆盖常见表格/伪表格场景
- 若现有 `DaisyDialog` 不足，先补基础 API
- 若 `content` 侧按钮/弹窗接口不一致，先做最小对齐

#### A-4 清理第一批高混用文件

首批建议目标：

- `src/options/components/sections/RestSection.ts`
- `src/options/components/controls/privacySettings.ts`
- `src/options/components/controls/VaultRouterView.ts`
- `src/options/components/sections/FragmentSection.ts`

目标不是大改业务，而是：

- 用统一组件替换同类手写控件
- 避免同文件内同时混用 3 套方式
- 先把“肉眼可见的混乱区”收掉

### 5.3 阶段验收

满足以下条件才算完成：

- 新增代码中不再出现新的基础控件手写实现
- `Options` 高混用区域第一批完成迁移
- `DaisyUIHelpers` 不再作为新代码首选入口
- 团队有明确的组件使用顺序说明
- 相关页面完成一轮真实浏览器回归，建议通过 Chrome DevTools MCP 执行：
  - `Options` 页面基本加载
  - 第一批迁移文件对应控件可见、可操作、可保存
  - 无新增明显 console error / 样式错位 / 焦点异常

---

## 6. 阶段 B：组件相关技术债与结构债修复

### 6.1 阶段目标

本阶段的目标是让后续“全量组件统一”不被结构问题反复打断。

如果不先做这些，后面组件统一会反复返工。

### 6.2 必做任务

#### B-1 继续拆解高耦合 orchestrator

重点目标：

- `src/options/app/bootstrap.ts`
- `src/options/components/controls/yamlConfigTable.ts`
- `src/content/clipper/components/dialog.ts`
- `src/content/reader/session.ts`
- `src/content/video/session.ts`

要求：

- 按职责拆分，不按行数硬切
- 至少拆出状态、DOM/presenter、副作用、平台/仓储访问
- 拆完后组件层只负责展示和交互，不继续承担过多编排逻辑

#### B-2 稳定 `Options` 状态链

重点文件：

- `src/options/state/StateManager.ts`
- `src/options/state/optionsStore.ts`

需要完成：

- 减少 `deepClone`
- 减少 `JSON.stringify` 比较
- 明确 state slice 边界
- 避免 UI 组件对全量大快照形成依赖

原因：

- 组件统一后，如果状态链仍然过重，性能问题和复杂度仍会把 UI 层拖回混乱状态

#### B-3 统一表单与 section 边界

重点目标：

- `BaseSection`
- `formSectionManager`
- `optionsFormAdapter`
- 各 `Section` 内的重复 form 结构

期望结果：

- 各 section 拥有一致的 header / body / actions / hint 组织方式
- 不再每个 section 自己发明一套表单排版和行为模式
- 为后续“视觉统一”和“代码统一”一起铺路

#### B-4 清理重复控件构造逻辑

重点关注：

- 重复的 checkbox label 包装
- 重复的 input cell / row / card 结构
- 重复的按钮行、操作区、状态提示区

要求：

- 共性结构应进入 shared presenter / helper / component
- 不再在每个 section 内复制 DOM 模板

### 6.3 阶段验收

- 关键大文件完成至少一轮拆分
- `Options` 状态链不再是全量深拷贝主导
- section / form 结构模式基本统一
- 组件迁移已不再频繁被结构问题阻塞
- 拆分后的主流程完成一轮回归验证：
  - 自动化测试通过
  - Chrome DevTools MCP 或真实浏览器手动验证通过
  - 无明显生命周期、解绑、重复渲染、焦点丢失问题

---

## 7. 阶段 C：性能优化前置治理

### 7.1 阶段目标

本阶段不追求把所有性能问题一次做完，而是先完成会直接影响组件统一质量的那部分优化。

### 7.2 必做任务

#### C-1 处理 `Options` 状态热路径

根据现有性能评估，优先级最高的是：

- `StateManager` 的重深拷贝与广播开销
- `optionsStore` 的重复规范化与快照广播

目标：

- 为组件统一后的 UI 提供可承载的状态基础
- 避免“视觉统一了，但交互更卡”

#### C-2 收敛内容脚本启动链

重点：

- `src/content/index.ts`
- `src/content/runtime/bootstrapRuntime.ts`

目标：

- 减少未使用模块的提前初始化
- 把 `Reader` / `Video` / `Prompt` / `Extractor` 的初始化门控做清晰

原因：

- 后续 `content` 侧组件统一时，会更容易明确“哪些 UI 是真正共用的，哪些只是场景组件”

#### C-3 标记高频 DOM 查询热点

重点：

- `Reader` markdown/export 路径
- `ArticleExtractor`
- `Video/Bilibili` 高频扫描路径

要求：

- 至少建立热点清单和 profile 基线
- 不必在本阶段一次性全修，但要避免后续 UI 统一时继续扩大热点

### 7.3 阶段验收

- `Options` 状态更新路径明显减重
- content 启动链有最小化门控
- 高频热点已被识别并形成后续优化顺序
- 对关键热点至少做一轮浏览器侧验证：
  - 使用 Chrome DevTools MCP 检查页面初始化、交互延迟、console、DOM 更新频率
  - 对优化前后体感、渲染次数或热点行为有基本对比记录

## 8. 阶段 D：全量组件统一

### 8.1 阶段目标

这是“工程美感”的主阶段。

目标不是只换个 class，而是让前端组件体系真正形成统一语言。

完成后应实现：

- 基础组件入口唯一
- 常见 UI 结构模式统一
- 同类交互行为一致
- `Options` 与 `content` 各自内部统一，并在共享层保持清晰边界

### 8.2 统一范围

#### D-1 `Options` 全量统一

需要覆盖：

- 全部 `sections/*`
- 全部 `controls/*`
- `layout/*`
- `shared/*`

目标：

- `Options` 不再存在大面积手写基础控件
- 每个 section 的标题、表单区、提示区、操作区风格一致
- 删除“某些页面像设计系统，某些页面像临时脚本”的割裂感

#### D-2 `content` 侧组件统一

需要覆盖：

- `clipper`
- `reader`
- `video`
- `supportPrompt`

目标：

- 统一弹窗、按钮、面板、列表项、状态 badge、操作栏
- 在共享视觉模式基础上保留场景差异，而不是每个模块自成体系

注意：

- `content` 与 `options` 不一定共享同一套实现类
- 但必须共享一套清晰的设计语义和命名语义

#### D-3 统一交互约定

必须统一的内容：

- 按钮语义：primary / secondary / ghost / danger
- 输入态与校验态
- 表格和伪表格布局
- modal / dialog 的 header/body/footer 结构
- 空状态、成功态、错误态、加载态
- 键盘交互和可访问性最小标准

#### D-4 统一设计 token 与样式来源

本阶段不要求一定把所有 CSS 物理合并成一个文件，但要求：

- 组件样式来源清晰
- token 使用一致
- 不再出现“同样的视觉语义在不同模块里靠不同 class 拼出来”

### 8.3 阶段验收

- `Options` 全量组件完成统一
- `content` 核心 UI 完成统一
- 基础交互语义统一
- 主要页面和核心流程视觉语言一致
- 全量组件统一后必须执行系统级手动回归，建议使用 Chrome DevTools MCP：
  - `Options` 全 section 浏览
  - `Reader` 主流程
  - `Video` 主流程
  - `Clipper` 主流程
  - 键盘操作、弹窗、焦点、拖拽、空状态、错误态

## 9. 阶段 E：最终收口与旧体系退役

### 9.1 阶段目标

这一阶段的目标是：让“统一”不只是表面现象，而是代码事实。

### 9.2 必做任务

#### E-1 删除旧 helper / 旧入口 / 兼容残留

重点处理：

- `DaisyUIHelpers.ts` 是否还能完全退役
- 各类临时 factory 是否还能删除
- 已迁移文件中的旧写法是否仍保留死代码

#### E-2 建立最终设计系统文档

需要沉淀：

- 组件目录结构
- 组件分层规则
- 组件命名规范
- 何时用 shared component，何时写场景组件
- 样式与 token 使用规范

#### E-3 建立持续守门

建议：

- 增加静态扫描，阻止新增手写基础控件
- 增加 UI review checklist
- 在 PR 审查中检查是否绕过统一组件入口

#### E-4 做最终统一验收

验收内容：

- 视觉一致性
- 可访问性
- 关键流程回归
- 组件 API 是否足够稳定
- 是否还存在明显“双轨”实现
- 手动测试是否真正覆盖真实浏览器行为
- 是否已使用 Chrome DevTools MCP 或等价方式完成最终人工回归

### 9.3 最终完成标准

当以下条件同时满足时，才能宣告“组件统一完成”：

- `Options` 与 `content` 两侧基础组件入口清晰且唯一
- 不再存在大面积基础控件手写重复实现
- 旧 helper / 旧工厂 / 旧兼容路径已退役或明确限缩
- 技术债与性能热点不再推动组件层重新分裂
- 新开发者可以按统一规则继续开发，而不会再次引入新体系
- 关键流程已完成自动化验证与真实浏览器手动验证
- 最终回归可通过 Chrome DevTools MCP 复现执行

## 11. 每阶段禁止事项

### 通用禁止事项

- 不做“全仓一次性 UI 重写”
- 不一边统一组件一边随意重做业务逻辑
- 不为了统一而制造新的抽象层级
- 不为了美感牺牲现有稳定性

### 阶段 A 禁止事项

- 不先去统一 `content` 全部 UI
- 不把“最小组件迁移”扩成“大规模视觉重设计”

### 阶段 B 禁止事项

- 不借拆文件之机重做整个架构
- 不并行拆太多 orchestrator，避免失控

### 阶段 C 禁止事项

- 不在组件接口未稳定前做大规模微优化
- 不为 benchmark 好看重写大量稳定逻辑

### 阶段 D 禁止事项

- 不混淆“共享组件”和“场景组件”
- 不强迫 `Options` 与 `content` 共用所有实现

### 阶段 E 禁止事项

- 不保留大量“以后可能用到”的兼容代码
- 不把最终收口变成新的长期拖尾区

## 12. 建议排期方式

如果按单人主导推进，建议采用：

- 第 1 段：阶段 A
- 第 2 段：阶段 B
- 第 3 段：阶段 C
- 第 4 段：阶段 D
- 第 5 段：阶段 E

如果按小团队推进，建议分工为：

- 一条线负责 `Options` 组件统一
- 一条线负责结构债与状态链
- 一条线负责 `content` 侧统一与性能热点

但前提是：

- shared 组件写入范围不能多人同时随意修改
- 必须先确定统一的组件规则和目录边界

## 13. 最终建议

对于追求工程美感的项目，这条路线的关键不是“尽快统一所有组件”，而是：

1. 先建立唯一方向
2. 再排除破坏统一的结构问题
3. 再做真正的全量统一
4. 最后把旧世界彻底退役

换句话说：

组件统一不应被当作最后的装饰，也不应被当作一开始就全面开战的大重构。

它应该是：

先小规模止血，
再中规模收口，
最后集中完成的一项主任务。

这才既能保住工程质量，也能真正做出你要的工程美感。
