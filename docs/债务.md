  七、新增发现的技术债务（2025-03-12 分析）

  13. 全局状态污染 (🔴 严重)

  现状: 4 个全局 window 属性用于状态管理

  ```typescript
  // content/index.ts:28-35
  declare global {
    interface Window {
      __aiobClipperInitialized?: boolean;
      __aiobVideoActive?: boolean;
      __aiobVideoController?: VideoSession;
      __aiobReaderActive?: boolean;
    }
  }
  ```

  问题:
  - 违背模块化原则，难以测试和追踪
  - 状态分散，无法保证单一真相源
  - 不同模块可能相互覆盖全局状态

  应对方案: 使用显式依赖传递或状态管理库替代全局属性

  ---
  14. 过度复杂的依赖注入运行时检查 (🔴 严重)

  现状: video/session.ts 中的依赖注入模式

  ```typescript
  let defaultVideoSessionDependencies: VideoSessionDependencies | null = null;

  function getDefaultVideoSessionDependencies(): VideoSessionDependencies {
    if (!defaultVideoSessionDependencies) {
      throw new Error('VideoSession dependencies have not been initialized');
    }
    return defaultVideoSessionDependencies;
  }
  ```

  问题:
  - 运行时检查增加认知复杂度
  - 调用链难以追踪，错误发生在运行时而非编译时
  - 违背 TypeScript 静态类型检查的初衷

  应对方案: 使用构造函数注入或工厂模式确保依赖在编译期可验证

  ---
  15. 类型系统配置债务 (🔴 严重)

  现状: tsconfig.base.json 配置过于宽松

  ```json
  {
    "strict": false,
    "isolatedModules": false
  }
  ```

  问题:
  - 基础配置关闭严格模式，需要单独 tsconfig.strict.json 做严格检查
  - 21 处显式 any 类型使用，类型安全无法保证
  - "isolatedModules": false 可能导致编译器无法检测跨文件类型问题

  应对方案:
  - 将 strict 设为 true 作为默认配置
  - 逐步消除所有 any 类型
  - 启用 isolatedModules 确保每个文件独立可编译
