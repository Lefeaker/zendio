## 1. 文档定位

这份文档不再沿用旧版本“把所有问题混成一个列表”的写法，而是按当前真实状态分为三类：

1. **已过时 / 已基本解决**：可以从主债务视角降级或归档
2. **仍然有效，但需要更新事实描述**：问题还在，但旧文档中的数字、文件名或严重程度已经不准确
3. **当前仍应优先处理**：而且其中一部分已经并入“目标架构迁移主线”

本文件的目标不是制造新的“债务文档债务”，而是给当前排期提供一个更接近代码真值的判断基线。

## 2. 当前总体判断

当前项目的真实状态可以概括为：

- 组件体系**不是没有建立**，而是已经进入“`Options` 侧较成熟、`content` 侧半统一、宿主差异尚未抽象完成”的阶段
- Repository / composition root / audit / 测试 / 构建基线都已经有明显进展
- 旧债务文档中有不少问题已经被解决或大幅缓解
- 当前最值得优先处理的，不是泛化的“全仓债务大扫除”，而是：
  - 组件统一相关问题
  - 与组件统一强耦合的结构债
  - 与宿主分层、交互契约、样式宿主、运行时依赖注入有关的问题

换句话说：

> 当前最优策略不是“先修所有债务再谈组件统一”，也不是“只谈组件统一不管债务”，而是**优先修复那些会阻碍目标架构落地的债务**。

## 5. 当前仍应优先处理的债务

以下问题仍然成立，而且从整体项目角度看，应优先进入执行视野。

其中一部分已经与“目标架构迁移主线”高度耦合，应该合并推进。

### 5.1 运行时依赖注入检查复杂（高优先级）

当前仍存在大量这种模式：

- `defaultVideoPromptDependencies`
- `initializeDefaultVideoPromptDependencies()`
- `getDefaultVideoSessionDependencies()`
- `initializeDefaultReaderSessionDependencies()`

主要位置：

- `src/content/video/prompt.ts`
- `src/content/video/session.ts`
- `src/content/reader/session.ts`
- `src/content/runtime/contentLazyRuntime.ts`

问题：

- 依赖是否初始化要到运行时才能发现
- 增加调用链复杂度
- 不利于清晰的宿主适配和组件分层
- 与“统一 host contract / feature 边界”强耦合

建议：

- 优先在目标架构迁移过程中，把这类 runtime default dependency 模式收口到：
  - 显式构造函数注入
  - 明确 factory 注入
  - 或 host adapter 统一装配

状态建议：

- **并入目标架构迁移主线**

---

### 5.2 类型系统默认基线仍偏宽松（高优先级）

当前实际情况：

- `tsconfig.base.json`：
  - `strict: false`
  - `isolatedModules: false`
- 仓库通过额外的 `tsconfig.strict.json` 做严格检查
- 当前代码仍可见残留：
  - `src/shared/schemas/options.schema.ts`：`z.any()`
  - `tests/**` 与少量平台 mock 中仍有受 `lint:type-any:ratchet` 约束的 assertion / `unknown` 债务

问题：

- 默认基线与理想目标架构不一致
- 组件 contract 统一后，如果默认类型系统仍宽松，会削弱架构收益

建议：

- 这项债务**仍然有效**
- 但不建议先于组件基础 contract 全面推进
- 更适合在基础原语和宿主分层稳定后，分阶段收紧

状态建议：

- **列为第二阶段高优先级债务**

---

### 5.3 直接 DOM 操作与业务逻辑混杂（高优先级）

当前粗略统计：

- `document.` 直接访问约 **284 处**

说明：

- DOM-first 架构本身不是问题
- 真正的问题是：
  - DOM 构建
  - 宿主细节
  - 业务状态
  - 样式绑定
  - 交互行为
  是否混在同一层

这项问题与以下事项强耦合：

- 组件统一
- host adapter
- style host
- dialog / panel contract
- feature 与 UI 壳层边界

建议：

- 不要把它当成“单纯减少 document 调用次数”的 KPI
- 应在目标架构迁移中，通过 `primitives/patterns/hosts` 分层来解决

状态建议：

- **并入目标架构迁移主线**

---

### 5.4 样式系统遗留债务（高优先级）

当前仍存在较重的遗留文件：

- `src/options/obsidian-clipper-style.css` — **83,072 bytes**
- `src/options/obsidian-hybrid-preview.html` — **60,755 bytes**
- `src/options/optionuicsssuggest.md` — **600,637 bytes**

问题：

- 容易让“正式设计系统入口”与“历史/预览/临时产物”混淆
- 与组件统一过程中要建立的 style host / token / layout contract 冲突
- 仓库噪音与维护成本高

建议：

- 与组件统一一起规划
- 明确：
  - 正式样式入口
  - 预览文件归属
  - 历史遗留归档策略

状态建议：

- **并入目标架构迁移主线**

---

### 5.5 组件相关的大文件和职责过载（高优先级）

这类文件不是“纯技术债”，而是会直接阻碍目标架构落地。

典型问题文件：

- `yamlConfigTableDom.ts`
- `RestSection.ts`
- `video/session.ts`
- `bilibiliPlatform.ts`
- `privacySettings.ts`
- `video/prompt.ts`

问题：

- 不利于抽 primitives / patterns / hosts
- 一改动就跨多层责任
- 回归验证成本高

建议：

- 不要以“行数”为唯一标准
- 按“责任密度”拆：
  - model
  - controller
  - presenter/view
  - host adapter
  - side effect

状态建议：

- **并入目标架构迁移主线**

## 6. 当前不建议优先处理的债务

以下问题仍然成立，但不建议抢占当前主线资源。

### 6.1 Tailwind 配置合并

虽然有 4 个 Tailwind config，但在多宿主扩展项目里，多配置并不天然错误。

建议：

- 等 `hosts` 层稳定后再决定是否合并
- 当前不要为了“配置整洁”先动它

---

### 6.2 测试工具统一

当前同时使用 Vitest / Playwright 并不一定是坏事，更多是合理分工。

建议：

- 保留“多工具分工”的现实策略
- 当前重点应放在测试契约和入口清晰，而不是工具绝对统一

---

### 6.3 i18n 代码生成化

这是有价值的中长期优化，但不应该优先于：

- 基础原语统一
- 宿主抽象
- 交互契约统一
- 样式宿主治理

---

### 6.4 配置文件大一统

`tsconfig` / `tailwind` / `vitest` 文件较多，确实会影响理解成本。

但当前阶段更重要的是：

- 先让组件架构与宿主边界清晰
- 再决定哪些配置是真重复，哪些是合理分场景

## 7. 从整体项目角度的优先级建议

## 7.1 第一优先级：组件统一主线 + 强耦合债务

应该优先一起推进的事项：

1. 组件基础原语统一（button / dialog / layout / host contract）
2. 直接 DOM 操作与 UI 壳层边界收紧
3. UI 相关大文件按职责拆分
4. 样式宿主与遗留样式体系治理
5. 运行时依赖注入复杂度降低

这些问题互相耦合，分开做会反复返工。

---

## 7.2 第二优先级：组件架构稳定后的结构收口

建议在第一优先级之后推进：

1. 默认 strict / isolatedModules 收紧
2. 配置与脚本体系整理
3. path / audit / build 规则进一步收口

---

## 7.3 第三优先级：中长期维护债

建议放在架构主线之后：

1. i18n 生成化
2. 文档归档与生命周期治理
3. preview / 遗留文件全面清仓
4. 非关键配置合并

---

## 10. 当前结论

- 承认组件统一已经成为主轴问题
