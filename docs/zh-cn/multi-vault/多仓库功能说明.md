# 多仓库功能说明

## 功能概述

All in Obsidian 现在支持配置多个 Obsidian 仓库，并通过智能路由规则自动选择目标仓库。这让你可以：

- 📦 **管理多个仓库**：工作笔记、个人笔记、学习笔记等分开管理
- 🔀 **智能路由**：根据网站域名、关键词或 URL 模式自动选择仓库
- ⚡ **灵活配置**：支持优先级、启用/禁用规则等高级功能

## 使用场景

### 场景 1：工作与生活分离

```
工作仓库：保存技术文档、项目资料
个人仓库：保存生活笔记、阅读摘录
```

**配置示例：**
- 规则 1：域名匹配 `github.com` → 工作仓库
- 规则 2：域名匹配 `stackoverflow.com` → 工作仓库
- 规则 3：关键词匹配 `旅行,美食,电影` → 个人仓库
- 默认：个人仓库

### 场景 2：按主题分类

```
技术仓库：编程、开发相关
产品仓库：产品设计、用户研究
学习仓库：课程笔记、学习资料
```

**配置示例：**
- 规则 1：关键词匹配 `编程,代码,开发` → 技术仓库
- 规则 2：关键词匹配 `产品,设计,用户` → 产品仓库
- 规则 3：关键词匹配 `课程,学习,教程` → 学习仓库

### 场景 3：按语言分类

```
中文仓库：中文内容
英文仓库：英文内容
日文仓库：日文内容
```

**配置示例：**
- 规则 1：域名匹配 `*.cn` → 中文仓库
- 规则 2：域名匹配 `*.jp` → 日文仓库
- 默认：英文仓库

## 核心概念

### 1. 仓库配置 (Vault Config)

每个仓库包含以下信息：

- **仓库名称**：用户友好的名称，如"工作笔记"
- **HTTPS URL**：Obsidian Local REST API 的 HTTPS 地址
- **HTTP URL**：Obsidian Local REST API 的 HTTP 地址（备用）
- **Vault 名称**：Obsidian 中的 vault 名称
- **API Key**：访问 API 的密钥
- **是否默认**：标记为默认仓库

### 2. 路由规则 (Routing Rule)

路由规则决定内容保存到哪个仓库：

- **规则类型**：
  - 域名匹配：根据网站域名
  - 关键词匹配：根据标题和内容中的关键词
  - URL 模式：根据 URL 正则表达式

- **匹配模式**：具体的匹配条件
- **目标仓库**：匹配成功后保存到的仓库
- **优先级**：数字越大优先级越高（0-100）
- **启用状态**：是否启用此规则
- **描述**：规则的说明文字

### 3. 路由逻辑

当剪藏内容时，系统按以下顺序选择仓库：

1. **按优先级匹配规则**：从高到低依次尝试匹配
2. **使用默认仓库**：如果没有规则匹配
3. **使用第一个仓库**：如果没有设置默认仓库

## 规则类型详解

### 域名匹配 (Domain Match)

匹配网页的域名。

**支持的模式：**
- 精确匹配：`example.com`
- 通配符匹配：`*.example.com`（匹配所有子域名）

**示例：**
```
模式：github.com
匹配：https://github.com/user/repo ✅
不匹配：https://gitlab.com/user/repo ❌

模式：*.google.com
匹配：https://mail.google.com ✅
匹配：https://drive.google.com ✅
不匹配：https://google.com ❌
```

### 关键词匹配 (Keyword Match)

在页面标题和内容中搜索关键词。

**格式：**
- 多个关键词用逗号分隔
- 任意一个关键词匹配即可
- 不区分大小写

**示例：**
```
模式：编程,代码,开发
匹配：标题包含"Python 编程入门" ✅
匹配：内容包含"代码示例" ✅
不匹配：标题和内容都不包含关键词 ❌
```

### URL 模式 (URL Pattern)

使用正则表达式匹配完整 URL。

**示例：**
```
模式：^https://github\.com/.*/(issues|pull)/\d+$
匹配：https://github.com/user/repo/issues/123 ✅
匹配：https://github.com/user/repo/pull/456 ✅
不匹配：https://github.com/user/repo ❌
```

## 配置步骤

### 1. 添加仓库

1. 打开设置页面
2. 点击"多仓库管理"
3. 点击"+ 添加仓库"
4. 填写仓库信息：
   - 仓库名称：如"工作笔记"
   - HTTPS URL：`https://127.0.0.1:27124/`
   - HTTP URL：`http://127.0.0.1:27123/`
   - Vault 名称：Obsidian 中的 vault 名称
   - API Key：从 Obsidian Local REST API 插件获取
5. 点击"保存"

### 2. 设置默认仓库

1. 在仓库列表中找到要设为默认的仓库
2. 点击"设为默认"按钮
3. 该仓库会显示"默认"标签

### 3. 添加路由规则

1. 在"路由规则"部分点击"+ 添加规则"
2. 选择规则类型：
   - 域名匹配：适合按网站分类
   - 关键词匹配：适合按内容主题分类
   - URL 模式：适合复杂的匹配需求
3. 填写匹配模式
4. 选择目标仓库
5. 设置优先级（可选，默认 10）
6. 添加规则描述（可选）
7. 确保"启用此规则"已勾选
8. 点击"保存"

### 4. 调整规则优先级

规则按优先级从高到低匹配。如果多个规则可能匹配同一内容，优先级高的规则会先被应用。

**建议：**
- 特定规则：优先级 50-100
- 一般规则：优先级 10-49
- 兜底规则：优先级 0-9

## 最佳实践

### 1. 规则设计原则

- **从特殊到一般**：特定规则优先级高，通用规则优先级低
- **避免冲突**：确保规则之间不会产生歧义
- **使用描述**：为每个规则添加清晰的描述
- **定期审查**：定期检查规则是否还需要

### 2. 性能优化

- **限制规则数量**：建议不超过 20 条规则
- **优先使用域名匹配**：比关键词匹配更快
- **避免复杂正则**：简单的正则表达式性能更好

### 3. 测试建议

添加新规则后，建议测试：

1. 打开符合规则的网页
2. 执行剪藏操作
3. 检查内容是否保存到正确的仓库
4. 查看浏览器控制台日志确认规则匹配情况

## 故障排查

### 问题 1：内容保存到错误的仓库

**可能原因：**
- 规则优先级设置不当
- 规则匹配模式过于宽泛
- 多个规则冲突

**解决方法：**
1. 检查规则优先级
2. 查看浏览器控制台日志，确认哪个规则被匹配
3. 调整规则的匹配模式或优先级

### 问题 2：规则不生效

**可能原因：**
- 规则未启用
- 匹配模式错误
- 目标仓库不存在

**解决方法：**
1. 确认规则的"启用"复选框已勾选
2. 检查匹配模式是否正确
3. 确认目标仓库配置正确且可访问

### 问题 3：无法连接到仓库

**可能原因：**
- Obsidian 未运行
- Local REST API 插件未启用
- API Key 错误
- URL 配置错误

**解决方法：**
1. 确保 Obsidian 正在运行
2. 确保 Local REST API 插件已启用
3. 检查 API Key 是否正确
4. 测试 HTTPS 和 HTTP URL 是否可访问

## 技术实现

### 核心类

- **`VaultRouter`**：路由器核心类
  - `selectVault(context)`：根据上下文选择仓库
  - `matchRule(rule, context)`：匹配单个规则
  - `validate()`：验证配置有效性

- **`VaultConfig`**：仓库配置
- **`RoutingRule`**：路由规则
- **`ClipContext`**：剪藏上下文（URL、域名、标题、内容等）

### 数据存储

配置存储在 `chrome.storage.sync` 中：

```typescript
{
  vaultRouter: {
    vaults: VaultConfig[],
    rules: RoutingRule[],
    defaultVaultId: string
  }
}
```

### 匹配算法

```
1. 获取所有启用的规则
2. 按优先级降序排序
3. 依次尝试匹配：
   - 域名匹配：比较域名
   - 关键词匹配：搜索标题和内容
   - URL 模式：正则表达式匹配
4. 返回第一个匹配的规则对应的仓库
5. 如果没有匹配，返回默认仓库
```
