# AllinOB 技术债务汇总

更新时间：2026-03-13

> 状态说明：本文件保留 2026-03-13 的结构审查结论，不再单独维护批次完成状态；2026-03-14 之后的执行真值以 `docs/251126-design-system-poc/DEBT-IMPLEMENTATION-ROADMAP.md` 为准。

本文档基于对 `AiiinOB` 当前代码结构的静态审查整理，面向交接、排期和债务治理讨论使用。

## 1. 结论摘要

当前项目的技术债务不是“代码完全失控”，而是典型的“多轮重构后保留了较多过渡层和兼容层”。

前端的主要问题集中在：

- `Options` 页正式入口已收口，但状态管理与持久化链路仍保留多层过渡代码
- 状态管理、持久化、UI 组件迁移仍有多层过渡代码
- 内容脚本和视频/阅读模块存在超大 orchestrator 文件
- 样式桥接和跨浏览器 Shadow DOM 兼容方案偏重

后端的主要问题集中在：

- 配置仓储接口和实现重复，存在两套配置读写模型
- DI 容器、平台服务定位和 legacy singleton 并存
- background service worker 入口职责过重
- listener/pipeline 文件承担过多编排逻辑
- 兼容旧配置、旧数据的迁移逻辑分散且缺少退役计划

## 2. 前端技术债务清单

### 2.1 P0

#### 1. `Options` 页新旧壳并存（2026-03-14 已完成重基线）

- 现状：
  - `src/options/index.ts -> src/options/app/bootstrap.ts` 已成为唯一正式页面启动链
  - `src/options/bootstrap.ts` 已降级为显式兼容入口，不再参与正式页面装配
- 风险：
  - 剩余风险已从“多入口”收敛为命名和局部 fallback 语义残留
  - 后续变更仍可能受 Options 仓储双轨影响
- 建议：
  - 维持唯一正式入口结论，不再把本项作为活跃批次
  - 后续仅在 Options 仓储统一或 orchestrator 拆分时顺手清理残余命名/兼容语义
- 预估工作量：已从活跃 P0 降为随批次收尾项

#### 2. Options 状态与持久化链路过度分层

- 现状：
  - `OptionsController`
  - `chromeOptionsPersistence`
  - `optionsStore`
  - `ChromeSyncOptionsRepository`
  - `ChromeOptionsRepository` / 其他仓储适配
  - 多层间存在读写、订阅、归一化、回写
- 风险：
  - 容易出现重复通知、重复保存、状态不同步
  - 出问题时定位困难
- 建议：
  - 收敛为一条“唯一数据主链”
  - UI 层只依赖一个 options store/repository 抽象
- 预估工作量：4-6 天

#### 3. 巨型 orchestrator 文件过多

- 典型文件：
  - `src/content/video/session.ts`
  - `src/content/index.ts`
  - `src/options/app/bootstrap.ts`
  - `src/options/components/controls/yamlConfigTable.ts`
- 风险：
  - 单文件承担状态、生命周期、副作用、平台适配、UI 协调等多个职责
  - 变更影响面大，单测补洞困难
- 建议：
  - 按“状态/副作用/UI/平台适配/数据访问”拆层
  - 避免只做机械切文件
- 预估工作量：6-10 天

### 2.2 P1

#### 4. 自定义状态管理器存在性能债

- 现状：
  - `src/options/state/StateManager.ts` 中大量 `deepClone`
  - 复杂字段比较依赖 `JSON.stringify`
- 风险：
  - section 和 options 规模继续增长后会带来不必要的性能损耗
- 建议：
  - 使用结构共享
  - 去掉 `JSON.stringify` 比较，改为更明确的字段变更判定
- 预估工作量：1-2 天

#### 5. UI 迁移未收尾，组件体系混用

- 现状：
  - DaisyUI / Tailwind / 原生 DOM / 手写交互控件混用
  - `LanguageSection`、`ReadingSection`、`RestSection` 等仍保留 Stage 3 TODO
- 风险：
  - 交互一致性差
  - 可访问性和复用能力受限
  - 后续样式回归成本高
- 建议：
  - 优先补齐 select/radio/table 三类基础抽象控件
  - 再逐步替换 section 内手写 DOM 控件
- 预估工作量：3-5 天

#### 6. Reader/Video 对话框重复实现

- 现状：
  - `src/content/reader/components/ReaderDialog.ts`
  - `src/content/video/components/VideoDialog.ts`
  - 两者结构和行为高度相似
- 风险：
  - 同类缺陷需要修两遍
  - 交互行为容易逐渐漂移
- 建议：
  - 抽取统一 list dialog / editable dialog presenter
- 预估工作量：2-3 天

### 2.3 P2

#### 7. Shadow DOM 样式桥接体系偏重

- 现状：
  - `src/content/shared/shadowStyleBridge.ts`
  - `src/content/clipper/shared/styleSheetManager.ts`
  - `src/content/shared/panels/styleSheetManager.ts`
  - 同时处理 adoptedStyleSheets、fallback style、跨浏览器 root 回放
- 风险：
  - 样式失效问题排查复杂
  - 兼容性 bug 定位成本高
- 建议：
  - 抽象统一 style host 策略
  - 合并 clipper/panel 的样式管理策略
- 预估工作量：2-4 天

#### 8. 遗留 feature flag 未完全清理

- 现状：
  - `src/content/reader/featureFlags.ts` 仍保留 `legacy | daisy`
- 风险：
  - 测试矩阵和阅读成本扩大
- 建议：
  - 若线上仅运行 `daisy`，删除 legacy 路径
- 预估工作量：0.5-1 天

## 3. 后端技术债务清单

注：此处“后端”指浏览器扩展中的 background、仓储、消息、平台适配、DI、REST 写入链路等后台职责代码。

### 3.1 P0

#### 1. 配置仓储双轨并存

- 现状：
  - `src/shared/repositories/IOptionsRepository.ts` 使用 `get/set/onChange`
  - `src/shared/interfaces/optionsRepository.ts` 使用 `load/save/snapshot/subscribe/reset`
  - 对应存在两套实现：
    - `src/infrastructure/repositories/ChromeOptionsRepository.ts`
    - `src/infrastructure/optionsRepository.ts`
- 风险：
  - 配置读写、订阅和默认值合并语义不统一
  - 测试替身、调用方和运行时行为容易割裂
- 建议：
  - 统一为一套仓储接口
  - 旧接口仅作为短期兼容层
- 预估工作量：4-6 天

#### 2. DI 体系重复且带 legacy 自动注册

- 现状：
  - `ServiceRegistry`
  - `ScopedServiceRegistry`
  - `RepositoryServiceContainer`
  - `platform/services.ts` 自动注册默认服务
  - `shared/di/index.ts` 中保留 `getLegacySingleton`
- 风险：
  - 依赖来源不透明
  - service worker 重启时状态一致性和测试隔离更难保证
- 建议：
  - 保留一套统一容器机制
  - 禁止运行时自动补注册
  - 平台服务改为显式装配
- 预估工作量：3-5 天

#### 3. background 入口职责过重

- 现状：
  - `src/background/index.ts` 同时承担：
    - 依赖引导
    - 监听器注册
    - usage stats 初始化
    - 试用版初始化
    - 首次安装引导
    - suspend 清理
- 风险：
  - service worker 上下文中职责过多，变更影响面大
- 建议：
  - 拆成 `bootstrap`、`lifecycle`、`trial`、`onboarding` 等模块
- 预估工作量：2-4 天

### 3.2 P1

#### 4. service worker 生命周期处理不稳

- 现状：
  - `src/background/index.ts` 使用 `setInterval` 周期检查试用状态
  - 再通过 `onSuspend` 做依赖清理
- 风险：
  - MV3 service worker 不适合依赖长期定时器
  - 容易出现本地可用、线上偶发失效
- 建议：
  - 改为事件触发或惰性检查
  - 仅在需要显示通知时做状态校验
- 预估工作量：1-2 天

#### 5. `contextMenus.ts` 已演化为巨型 orchestration

- 现状：
  - `src/background/listeners/contextMenus.ts` 401 行
  - 混合 URL 判断、注入策略、菜单状态、frame 决策、tab 缓存
- 风险：
  - 文件理解和维护成本过高
  - 很难做针对性测试
- 建议：
  - 拆出：
    - `videoUrlPolicy`
    - `menuStateService`
    - `contentInjectionCoordinator`
    - `tabVideoStateCache`
- 预估工作量：2-4 天

#### 6. `clipPipeline.ts` 和相关链路边界不清

- 现状：
  - `src/background/pipelines/clipPipeline.ts` 同时处理：
    - payload 归一化
    - 业务异常封装
    - 通知
    - content UI 回传
  - `src/background/application/clipProcessor.ts` 再承担写库、分类、路由
- 风险：
  - pipeline/application/service 间职责边界不够清晰
- 建议：
  - pipeline 只做流程编排
  - 通知、UI 回传、归一化拆成独立协作者
- 预估工作量：2-3 天

#### 7. `configService.ts` 是半成品服务仍在主路径

- 现状：
  - `isValidStoredOptions()` 基本恒真
  - 文件中保留 future use / placeholder 事件逻辑
- 风险：
  - 容易误导后来维护者把它当作正式配置入口
- 建议：
  - 要么删除
  - 要么基于现有 zod schema 重写为正式服务
- 预估工作量：1 天

### 3.3 P2

#### 8. 基础设施层与应用层职责混杂

- 现状：
  - `src/infrastructure/restClient.ts` 中直接做：
    - 业务日志
    - 错误处理上报
    - 聚合业务语义错误
- 风险：
  - 基础设施层不可复用
  - 单元测试更重
- 建议：
  - 基础设施层只抛 typed error
  - 聚合、通知、上报放回 pipeline/service
- 预估工作量：2-3 天

#### 9. 兼容旧数据逻辑分散

- 现状：
  - `usageStats`
  - `vault-router`
  - `taxonomyMigration`
  - `optionsMerger`
  - 多处都承担 legacy 兼容职责
- 风险：
  - 迁移窗口长期存在
  - 难以确认何时可以删除旧逻辑
- 建议：
  - 集中迁移策略到统一 migration layer
  - 明确版本和退役时间
- 预估工作量：2-3 天

#### 10. 后端边界类型仍不够硬

- 现状：
  - 若干 schema 中仍有 `z.any()`
  - 部分领域字段直接做 type cast
- 风险：
  - 跨进程消息和持久化边界仍可能吞掉错误结构
- 建议：
  - 优先强化消息边界和持久化边界 schema
- 预估工作量：1-2 天

## 5. 推荐治理顺序

### 第一阶段：先清主链

1. 统一后端 options repository
2. 收敛前端 options 持久化/状态主链
3. 收敛 DI 和平台服务装配方式

### 第二阶段：拆大文件和大编排层

1. 拆 `src/background/index.ts`
2. 拆 `src/background/listeners/contextMenus.ts`
3. 拆 `src/content/index.ts`
4. 拆 `src/content/video/session.ts`
5. 拆 `src/options/components/controls/yamlConfigTable.ts`

### 第三阶段：清过渡层

1. 删除 `Options` 新旧壳并存逻辑
2. 删除遗留 feature flag
3. 收口 legacy migration 逻辑
4. 删除半成品服务和占位实现

### 第四阶段：做质量收尾

1. 补严格 schema
2. 补针对性单元测试
3. 纯化基础设施层
4. 补齐 UI 基础控件抽象

## 6. 建议交付方式

如果需要把本清单交给执行方，建议按以下方式拆成任务：

- 每项债务单独建卡
- 卡片字段至少包含：
  - 问题描述
  - 影响范围
  - 改造目标
  - 涉及文件
  - 回归测试要求
  - 是否允许分阶段落地

建议第一批只立 P0 和 P1，不要一次性并行处理所有 P2。
